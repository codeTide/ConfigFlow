# برنامه مرحله‌ای حذف کامل `type_id`

این سند مسیر اجرایی حذف `type_id` از کل پروژه را به‌صورت کم‌ریسک و مرحله‌ای تعریف می‌کند.

## هدف
- حذف وابستگی منطقی و دیتابیسی به `type_id` در فلو خرید، مدیریت سرویس، موجودی و پرداخت.
- استانداردسازی مسیرها بر پایه `service` و `service_tariff`.

## وضعیت فعلی (خلاصه)
- `type_id` در جدول‌های اصلی مثل `service` و `configs` وجود دارد.
- بخشی از stateها و کوئری‌ها هنوز با `type_id` کار می‌کنند.
- بخش ادمین در بعضی منوها بر مبنای type طراحی شده است.

---

## فاز 0 — آماده‌سازی و ممیزی (بدون تغییر رفتاری)
1. استخراج لیست کامل استفاده‌ها:
   - ستون دیتابیس (`service.type_id`, `configs.type_id`, `packages.type_id` در صورت وجود).
   - stateهای بات (`buy.await_type` و payloadهایی که `type_id` حمل می‌کنند).
   - متون UI مرتبط با «نوع سرویس».
2. دسته‌بندی وابستگی‌ها:
   - **Run-time critical** (مسیر خرید/پرداخت فعال)
   - **Admin-only** (منوها/گزارش‌های مدیریتی)
   - **Legacy/Dead paths**
3. تعیین معیار موفقیت هر فاز + پلن rollback.

خروجی فاز: چک‌لیست دقیق فایل/تابع/مهاجرت.

---

## فاز 1 — حذف وابستگی منطقی از فلو کاربر (Compatibility Mode)
1. حذف کامل state مبتنی بر type در خرید کاربر:
   - حذف `buy.await_type` و مسیرهای برگشت وابسته.
   - تمام navigationهای خرید فقط با `service_id` و `tariff_id`.
2. حذف فیلد `type_id` از payload stateهای user-flow.
3. جایگزینی Queryها:
   - هرجا `list...ByType` در مسیر خرید است، تبدیل به list سراسری سرویس‌های فعال با تعرفه فعال.
4. حفظ سازگاری دیتابیس در این فاز:
   - ستون هنوز حذف نشود؛ فقط در کد خوانده/استفاده نشود.

خروجی فاز: کاربر نهایی هیچ نقطه‌ای type-based نبیند و کد runtime هم type-independent شود.

---

## فاز 2 — بازطراحی ادمین از Type-centric به Service-centric
1. منوهای ادمین «انواع/پکیج‌ها» به «سرویس‌ها/تعرفه‌ها» تبدیل شود.
2. ایجاد/ویرایش سرویس بدون انتخاب type.
3. فهرست‌ها و فیلترها بر اساس:
   - سرویس
   - تعرفه
   - وضعیت فعال/غیرفعال
4. در صورت نیاز برای طبقه‌بندی:
   - جایگزین `type_id` با `service.group_key` (اختیاری و متنی) یا tag.

خروجی فاز: پنل مدیریت هم عملاً بدون نیاز به type کار کند.

---

## فاز 3 — Migration دیتابیس (دو مرحله‌ای برای ریسک کم)

### 3.1 مهاجرت نرم (Expand/Backfill/Read-Switch)
1. حذف FKها و indexهای وابسته به `type_id` (در صورتی که وجود دارند) به شکل کنترل‌شده.
2. حذف dependency از کد (باید قبل از drop کامل انجام شده باشد).
3. اجرای smoke migration روی دیتای staging.

### 3.2 مهاجرت سخت (Drop)
1. Drop ستون‌ها:
   - `service.type_id`
   - `configs.type_id`
   - هر ستون type-based دیگر که واقعاً بلااستفاده است.
2. حذف جدول `config_types` فقط وقتی:
   - هیچ کد runtime/admin دیگر به آن وابسته نباشد.
3. پاکسازی migrationهای transitional و مستندسازی نهایی.

خروجی فاز: schema نهایی بدون `type_id`.

---

## فاز 4 — پاکسازی نهایی کد
1. حذف متدهای دیتابیسی مرتبط با type:
   - `getActiveTypes`, `countServicesWithTariffsByType`, `listActiveServicesByType`, ...
2. حذف keyهای ترجمه مربوط به «نوع سرویس».
3. حذف state names و branchهای routing وابسته به type.
4. به‌روزرسانی README و راهنمای ادمین.

---

## تست و اعتبارسنجی پیشنهادی در هر فاز
- تست دستی سناریوهای اصلی:
  1. خرید سرویس fixed
  2. خرید سرویس per_gb
  3. برگشت/منوی اصلی در همه stateها
  4. پرداخت کیف پول + درگاه‌ها
  5. تحویل سفارش و وضعیت pending/approved
- تست migration روی dump واقعی staging.
- افزودن تست رگرسیون برای state machine خرید.

---

## ریسک‌ها و کنترل ریسک
- **ریسک ناسازگاری پرداخت**: payloadهای قدیمی ممکن است `type_id` داشته باشند.
  - راهکار: reader tolerant تا یک نسخه (نادیده گرفتن `type_id` اگر موجود بود).
- **ریسک در ادمین**: مسیرهای قدیمی type-based از کار می‌افتند.
  - راهکار: rollout با feature flag ادمین.
- **ریسک migration**: خطا در drop به دلیل FK/index.
  - راهکار: اجرای dry-run و migration idempotent.

---

## ترتیب اجرای پیشنهادی (واقعی و عملی)
1. فاز 0 (ممیزی) — 0.5 تا 1 روز
2. فاز 1 (user-flow) — 1 تا 2 روز
3. فاز 2 (admin-flow) — 1 تا 3 روز
4. فاز 3 (db migration) — 0.5 تا 1 روز
5. فاز 4 (cleanup) — 0.5 روز

> پیشنهاد: اول فاز 0 و 1 را همین الان شروع کنیم تا سریع به نسخه‌ای برسیم که runtime کاملاً بدون type باشد؛ سپس ادمین و دیتابیس را drop نهایی کنیم.
